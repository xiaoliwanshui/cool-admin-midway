## 启动失败

### 端口占用
当启动 `cool-admin-midway` 时，若默认端口 8001 已被占用，系统将无法正常启动。

**排查步骤**：
1. 检查当前端口使用情况：
   - Windows: `netstat -ano | findstr :8001`
   - Linux/macOS: `lsof -i :8001`
2. 查看项目日志是否提示类似 `Port 8001 is occupied` 的警告信息。
3. 确认是否有其他进程（如旧实例、Docker 容器）占用了该端口。

**解决方案**：
- 修改配置文件 `src/config/config.local.ts` 中的端口设置（如未显式指定，则默认为 8001）。
- 或等待系统自动查找可用端口（从 8001 开始递增尝试，最多到 8010）。
- 手动终止占用端口的进程或更改应用监听端口。

```typescript
// 示例：在 config.local.ts 中指定端口
export default {
  koa: {
    port: 8002 // 更改端口
  },
  // ... 其他配置
}
```

### 数据库连接异常
启动时若无法连接数据库，会导致服务初始化失败。

**排查步骤**：
1. 检查 `src/config/config.local.ts` 中数据库配置是否正确（host、port、username、password、database）。
2. 确认数据库服务（MySQL/PostgreSQL/SQLite）是否已启动并可访问。
3. 检查 Redis 配置及服务状态，因部分功能依赖 Redis（如任务队列 Bull）。
4. 查看控制台错误日志，识别具体错误类型（如 `ECONNREFUSED`、`ENOTFOUND`）。

**解决方案**：
- 使用 Docker 启动数据库环境：`docker-compose up -d`
- 确保数据库用户名、密码、数据库名与配置一致。
- 若使用远程数据库，请检查防火墙和网络策略。
- 开启 TypeORM 的 `logging: true` 以输出 SQL 日志辅助诊断。

## API 返回 403 权限不足

### 问题原因
API 接口返回 403 表示请求被拒绝，通常由权限校验中间件拦截所致。

**常见场景**：
- 请求未携带有效的 `Authorization` 头部。
- 用户角色无权访问该接口。
- 接口被标记为需权限访问，但调用方未通过认证。

**排查步骤**：
1. 检查请求是否包含 JWT Token，格式为 `Bearer <token>`。
2. 确认用户登录后分配的角色是否具备对应接口权限。
3. 查看 `base/authority.ts` 中的权限控制逻辑。
4. 检查接口是否被错误地标记为需权限访问。

**解决方案**：
- 确保登录成功并正确传递 `Authorization` 头。
- 在管理后台为用户角色分配相应菜单权限。
- 若为公共接口，可在控制器方法上添加 `ignoreToken: true` 配置。

```typescript
@CoolController({
  api: ['info'],
  entity: DemoAppGoodsEntity,
  ignoreToken: true // 允许未登录访问
})
```

## JWT 认证失效

### 问题表现
用户登录后短时间内提示“登录失效”，或刷新页面需重新登录。

**根本原因**：
- JWT Token 过期时间设置过短。
- `isRefresh` 标志位被错误识别。
- Token 被服务器主动注销或缓存失效。

**排查步骤**：
1. 检查 `src/modules/base/config.ts` 和 `src/modules/user/config.ts` 中的 JWT 配置：
   - `expire`: Token 有效时间（默认 2 小时）
   - `refreshExpire`: 刷新 Token 有效时间（默认 15 天）
2. 查看中间件 `user/app.ts` 是否正确解析 Token。
3. 确认服务端是否调用了 `midwayCache.set` 存储 Token。

**解决方案**：
- 调整 JWT 过期时间以适应业务需求。
- 确保 `generateToken` 方法正确设置 `isRefresh` 字段。
- 检查 Redis 缓存是否正常运行，避免 Token 缓存丢失。

```typescript
// 示例：JWT 配置
jwt: {
  expire: 60 * 60 * 24,        // 1天
  refreshExpire: 60 * 60 * 24 * 7, // 7天
  secret: 'your-secret-key'
}
```

## Swagger 文档未生成

### 问题现象
访问 `/swagger` 页面空白或提示“Eps未开启”。

**原因分析**：
- `cool.eps` 配置未启用。
- Swagger 构建服务未正确初始化。
- 控制器未使用 `@CoolController` 装饰器暴露接口。

**排查步骤**：
1. 检查 `src/config/config.local.ts` 中 `cool.eps` 是否为 `true`。
2. 查看 `src/modules/swagger/builder.ts` 是否成功执行 `init()`。
3. 确认控制器类是否继承 `BaseController` 并使用 `@CoolController`。
4. 检查浏览器开发者工具中 `/swagger/json` 接口是否返回有效数据。

**解决方案**：
- 确保配置文件中启用 EPS：
```typescript
cool: {
  eps: true // 必须为 true
}
```
- 确保模块正确注册，Swagger 事件监听器触发：
```typescript
@CoolEvent()
export class SwaggerAppEvent {
  @Event('onServerReady')
  async onServerReady() {
    this.swaggerBuilder.init(); // 确保调用
  }
}
```

## 任务队列不执行

### 问题描述
定时任务或队列任务未按预期执行，无日志输出。

**可能原因**：
- 任务类型配置错误（`bull` vs `local`）。
- Redis 未启用或版本过低（Bull 要求 Redis >= 6.x）。
- 任务中间件拦截导致无法添加任务。
- 任务服务未初始化。

**排查步骤**：
1. 检查 `TaskInfoService.initType()` 是否正确识别任务类型。
2. 查看 `task/middleware/task.ts` 是否抛出异常（如 Redis 未连接）。
3. 确认 `taskBullService.initTask()` 是否被调用。
4. 检查数据库 `task_info` 表中任务状态是否为运行中。

**解决方案**：
- 确保 Redis 正常运行且版本符合要求。
- 若使用 `bull` 类型，确认 `CoolQueueHandle` 可注入。
- 手动调用 `taskInfoService.initTask()` 初始化任务。
- 检查任务类是否继承 `BaseCoolQueue` 并实现 `data(job, done)` 方法。

```typescript
@CoolQueue()
export class DemoCommQueue extends BaseCoolQueue {
  async data(job: any, done: any): Promise<void> {
    console.log('任务执行:', job.data);
    done(); // 必须调用 done()
  }
}
```

## 网络错误处理策略

引用 `network-error-handling.md` 中的网络错误处理机制，用于优雅应对数据库连接超时、第三方服务不可用等情况。

### 核心机制
- **NetworkErrorHandler 服务**：集中处理各类网络异常。
- **智能重试机制**：支持指数退避算法（1s, 2s, 4s）。
- **错误分类识别**：区分 DNS 解析失败、超时、连接拒绝等。
- **详细日志记录**：便于定位问题根源。

### 支持的错误类型
| 错误类型 | 描述 | 解决方案 |
|--------|------|---------|
| `ENOTFOUND` | DNS 解析失败 | 检查域名配置，更换 DNS |
| `ETIMEDOUT` | 请求超时 | 增加超时时间，检查网络 |
| `ECONNREFUSED` | 连接被拒绝 | 检查目标服务是否运行 |

### 使用示例
```typescript
const result = await this.networkErrorHandler.requestWithRetry(
  {
    url: 'https://api.example.com/data',
    method: 'GET',
    timeout: 30000
  },
  3, // 最多重试3次
  2000 // 初始延迟2秒
);
```

## 调试技巧与日志输出

### 启用详细日志
- 在 `config.local.ts` 中开启 TypeORM 日志：
```typescript
logging: true
```
- 使用 `console.log` 输出关键变量和流程信息。
- 查看 Redis 日志确认队列状态。
- 检查数据库日志（如 MySQL slow query log）。

### 断点调试
- 使用 VS Code 配置 Node.js 调试环境。
- 在关键服务方法（如 `login`、`data`）中设置断点。
- 利用 `debugger` 语句暂停执行。

### 日志建议
- 所有网络请求应记录 URL、参数、响应时间。
- 任务执行前后记录状态变化。
- 异常捕获时输出完整堆栈信息。
