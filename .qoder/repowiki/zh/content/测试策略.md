## 引言
本测试策略文档旨在为 `cool-admin-midway` 项目建立系统化、可维护的测试体系。通过采用 Jest 作为单元测试框架，结合 TypeORM 测试数据库，确保服务层逻辑的正确性与稳定性。文档详细说明现有测试用例的设计思路、测试编写规范、配置项含义，并推荐 TDD 开发流程与 E2E 测试集成方案，全面提升代码质量，防止回归错误。

## 测试框架与环境配置

### Jest 配置解析
项目根目录下的 `jest.config.js` 文件定义了测试运行的核心配置：

```js
module.exports = {
  preset: 'ts-jest',
  testEnvironment: 'node',
  testPathIgnorePatterns: ['<rootDir>/test/fixtures'],
  coveragePathIgnorePatterns: ['<rootDir>/test/'],
};
```

- **preset: 'ts-jest'**：使用 ts-jest 预设，支持 TypeScript 文件的自动编译与测试。
- **testEnvironment: 'node'**：指定测试运行在 Node.js 环境中，适用于后端服务测试。
- **testPathIgnorePatterns**：忽略 `test/fixtures` 目录下的文件，避免测试数据干扰。
- **coveragePathIgnorePatterns**：生成覆盖率报告时忽略 `test/` 目录，聚焦业务代码。

该配置确保测试环境与项目技术栈（TypeScript + Midway）无缝集成。

### 唯一键冲突处理测试（duplicate-key-test.ts）

2. **键值提取**：`extractDuplicateKey()` 方法能否从错误消息中解析出冲突的具体值（如视频标题）。

测试覆盖了多种数据库错误场景，确保系统在遇到重复数据时能优雅降级而非崩溃。

## 测试用例编写规范

### Controller 层测试
建议使用 **Supertest** 模拟 HTTP 请求，验证接口的响应状态码、数据结构与业务逻辑。示例结构如下：

```ts
import request from 'supertest';
import { createApp, closeApp } from './bootstrap';

describe('VideoController', () => {
  let app;
  beforeAll(async () => {
    app = await createApp();
  });

  afterAll(async () => {
    await closeApp();
  });

  it('应成功获取视频列表', async () => {
    const response = await request(app.callback()).get('/api/videos');
    expect(response.status).toBe(200);
    expect(response.body.data).toBeArray();
  });
});
```

### Service 层测试
Service 层测试应聚焦业务逻辑，通过 **Mock 依赖服务** 隔离外部影响。例如：

- Mock 数据库操作（使用 TypeORM 的 Repository Mock）
- Mock 第三方 API 调用（使用 jest.fn() 模拟返回值）
- 验证核心方法的输入输出与异常处理

## 覆盖率与质量保障

### 覆盖率报告生成
Jest 配置已启用覆盖率检查。运行 `jest --coverage` 可生成详细的覆盖率报告，包含：

- **语句覆盖率（Statements）**：代码行执行比例
- **分支覆盖率（Branches）**：if/else 等分支执行情况
- **函数覆盖率（Functions）**：函数调用比例
- **行覆盖率（Lines）**：代码行覆盖情况

建议将覆盖率阈值纳入 CI/CD 流程，例如要求分支覆盖率不低于 80%。

### 断言与异常处理验证
测试用例应包含对正常路径与异常路径的完整断言：

- 使用 `expect(...).toThrow()` 验证预期异常
- 使用 `try-catch` 捕获异步错误并断言错误类型
- 验证错误日志是否按预期输出

## 端到端测试集成方案

### 使用 Supertest 进行 API 测试
推荐在 `test/e2e/` 目录下建立端到端测试套件，使用 Supertest 与真实数据库（或测试数据库）交互，验证整个请求-响应流程。

```ts
import request from 'supertest';
import { Application } from '@midwayjs/koa';
import { createApp } from '../bootstrap';

describe('E2E: Video Collection API', () => {
  let app: Application;

  beforeAll(async () => {
    app = await createApp();
  });

  afterAll(async () => {
    await app.close();
  });

  it('应成功创建并查询视频', async () => {
    // 创建视频
    const createRes = await request(app.callback())
      .post('/api/videos')
      .send({ title: '测试视频' });
    expect(createRes.status).toBe(200);

    // 查询视频
    const getRes = await request(app.callback())
      .get('/api/videos')
      .query({ title: '测试视频' });
    expect(getRes.body.data.length).toBeGreaterThan(0);
  });
});
```

此方案可有效防止接口变更导致的集成问题。

## 测试驱动开发（TDD）建议

建议团队采用 **测试驱动开发（TDD）** 流程：

1. **红**：先编写失败的测试用例，明确功能需求。
2. **绿**：编写最简代码使测试通过。
3. **重构**：优化代码结构，确保测试仍通过。

TDD 能显著提升代码设计质量，减少后期调试成本，并形成天然的文档。

## 结论

`cool-admin-midway` 项目已具备良好的测试基础，通过 Jest + TypeORM 的组合，能够有效验证服务层核心逻辑。现有测试用例覆盖了唯一键冲突、网络异常等关键容错场景。建议进一步完善 Controller 层的 E2E 测试，建立 CI/CD 中的覆盖率门禁，并推广 TDD 实践，以持续保障代码质量，降低系统风险。
